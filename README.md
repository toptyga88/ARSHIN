## 💡 О проекте

**АРШИН** — это пайплайн для автоматического сбора и анализа данных о поверках счётчиков воды и тепла из государственного реестра [ФГИС Аршин](https://fgis.gost.ru/fundmetrology/cm/icsbase). Заменяет ручной Excel-справочник: данные обновляются автоматически, аналитика доступна в реальном времени через DataLens-дашборды.

### Зачем

| 🔄 | **Данные обновляются ежедневно** — ручной сбор устаревает за неделю |
|----|---|
| 🏷️ | **23 бренда конкурентов** — у каждого десятки моделей и сотни поверителей |
| 📈 | **Динамика по месяцам** — кто растёт, кто падает, где и когда |

### Принцип проектирования

Сначала данные — потом код. Перед написанием любого скрипта мы разбираемся **какие данные нам отдаёт API**, что каждое поле означает в предметной области, и какая структура хранения нам нужна. Только потом — алгоритмы и логика. Этот README отражает тот же порядок.

## 🌐 Источник данных

Данные берутся из публичного API государственного реестра ФГИС Аршин. Документации нет — все эндпоинты и поля найдены через DevTools браузера. Базовый URL:

```
https://fgis.gost.ru/fundmetrology/eapi
```

В проекте используются два эндпоинта семейства: **MIT** (справочник моделей измерительных приборов) и **VRI** (реестр поверок этих приборов). Подробно про каждый ниже.

---

### 1. MIT — справочник моделей

**MIT** (Государственный реестр Средств Измерений) хранит карточки **зарегистрированных моделей** измерительных приборов. Это «паспорт» модели: кто её разработал, какие у неё характеристики, какая методика поверки, какой срок действия типа.

#### 1.1. Поиск моделей

Запрос:

```
GET /mit?search=<подстрока>&rows=<число>&start=<offset>
```

Параметры запроса:

| Параметр | Тип | Смысл |
|----------|-----|-------|
| `search` | string | подстрока для поиска по полю «производитель/разработчик». Например `search=Декаст` найдёт все модели где «Декаст» упоминается в производителях |
| `rows` | integer | сколько записей вернуть за один запрос (максимум 100) |
| `start` | integer | offset для пагинации |

Что возвращает (`result.items[]`):

| Поле | Тип | Смысл в предметной области |
|------|-----|---------------------------|
| `number` | string | **рег.номер модели в реестре**, например `12345-21`. Это основной идентификатор модели, по нему сшивается всё дальше |
| `title` | string | официальное название модели, например `Счетчики воды крыльчатые СВК` |
| `manufacturers` | string | производитель/разработчик из карточки модели. Может содержать несколько организаций через запятую |
| `mit_uuid` | string (UUID) | внутренний идентификатор записи в реестре. Нужен для запроса детальной карточки (см. 1.2) |
| `notation` | string | условное обозначение типа СИ, иногда содержит диапазоны диаметров |
| `mitypes` | array | список типов измерений к которым относится модель (счётчик воды, теплоэнергии и т.д.) |
| `valid_date` | string | срок действия свидетельства о регистрации типа |

Используется для: получения списка всех моделей конкретного бренда (Шаг 0, Шаг 1).

#### 1.2. Детальная карточка модели

Запрос:

```
GET /mit/<mit_uuid>
```

Что возвращает дополнительно к основному списку:

| Поле | Тип | Смысл в предметной области |
|------|-----|---------------------------|
| `country` | string | страна производителя |
| `city` | string | город производителя |
| `methodics` | array | список **PDF-методик поверки** с прямыми ссылками на скачивание. В этих PDF производитель указывает все возможные диаметры/модификации модели — главный источник данных для классификации |
| `description` | string | описание модели — иногда содержит явное указание категории («бытовые», «промышленные») |

Используется для: получения страны/города (Шаг 0), извлечения диаметров из PDF (Шаг 2).

### 2. VRI — реестр поверок

**VRI** (Verification Registry) — это **реестр фактически совершённых поверок** приборов в эксплуатации. Каждая запись = одна поверка одного экземпляра прибора.

Чтобы понять разницу: MIT — это **что бывает** (модель зарегистрирована, имеет такие характеристики). VRI — это **что произошло** (такого-то числа такой-то экземпляр такой-то модели поверила такая-то организация).

#### 2.1. Поиск поверок

Запрос:

```
GET /vri?mit_number=<X>&year=<Y>&rows=<N>&start=<offset>
        [&org_title=...]
        [&verification_date_start=YYYY-MM-DD]
        [&verification_date_end=YYYY-MM-DD]
```

Параметры запроса:

| Параметр | Тип | Обязат. | Смысл |
|----------|-----|---------|-------|
| `mit_number` | string | да | рег.номер модели — те самые `12345-21` из MIT |
| `year` | integer | да | год поверок |
| `rows` | integer | нет | сколько записей за запрос (макс 100) |
| `start` | integer | нет | offset для пагинации. **Жёсткий лимит API: start ≤ 99 999** |
| `org_title` | string | нет | фильтр по конкретному поверителю |
| `verification_date_start` | date | нет | начало периода (только формат `YYYY-MM-DD`, `DD.MM.YYYY` не работает) |
| `verification_date_end` | date | нет | конец периода |

Что возвращает на верхнем уровне:

| Поле | Тип | Смысл |
|------|-----|-------|
| `result.count` | integer | **общее число поверок** удовлетворяющих фильтру. Самое ценное поле — даёт точное количество без необходимости тянуть все записи |
| `result.items` | array | первая страница самих записей |

Что возвращает в каждой записи (`result.items[]`):

| Поле | Тип | Смысл в предметной области |
|------|-----|---------------------------|
| `mit_number` | string | рег.номер модели — связь с MIT |
| `mit_title` | string | название модели как в MIT |
| `mi_modification` | string | **конкретная модификация поверенного экземпляра**, обычно содержит диаметр. Примеры: `СВК-15`, `СВМ Ду 25`, `СГВЭ-20 ALEM`. Главный источник информации о диаметре отдельной поверки |
| `mi_number` | string | заводской номер экземпляра прибора |
| `verification_date` | date | дата поверки в формате `DD.MM.YYYY` |
| `valid_date` | date | дата окончания действия поверки (срок следующей) |
| `org_title` | string | **название организации-поверителя**. Это и есть тот кто реально провёл поверку — может быть либо сам производитель (первичная поверка на заводе), либо аккредитованная лаборатория (периодическая поверка в эксплуатации) |
| `vriType` | string | **тип поверки**: `"1"` = первичная (на заводе перед отгрузкой), `"2"` = периодическая (через несколько лет в эксплуатации). Основа алгоритма определения производителя |
| `applicability` | string | пригодность (`Пригоден` / `Непригоден`) — для нас всегда фильтр на `Пригоден` |
| `result_docnum` | string | номер свидетельства о поверке |

Используется для: определения производителей через долю `vriType=1` (Шаг 1), подсчёта объёмов по месяцам и категориям (Шаг 4).

#### 2.2. Детальная карточка поверки

Запрос:

```
GET /vri/<id>
```

Используется только как ping-запрос для проверки доступности API. В пайплайне на содержимое мы не опираемся.

### Особенности API и грабли

| 🐛 | **HTTP 500 при фильтре по дате** — `verification_date_start/end` нестабильно работает на больших объёмах данных, периодически отдаёт ошибку. Решение: страховочная выборка за весь год с последующим распределением по месяцам |
|----|---|
| 📏 | **Жёсткий лимит пагинации** — параметр `start ≤ 99 999`. Если у поверителя более 100 000 поверок за год — полностью пройти список нельзя. Решение: умная выборка из 6 точек × 15 записей по списку |
| 🌍 | **TLS-соединение требует свежие сертификаты** — стандартный openssl у Python 3.10+ иногда не договаривается с реестром. Используем `certifi` для CA и кастомный TLSAdapter |
| 🤔 | **Нет признака «это производитель»** — в реестре никто специально не помечает кто производитель, а кто эксплуатант. Поэтому в Шаге 1 определяем эвристически по доле `vriType=1` |
| 📅 | **Формат дат строгий** — `YYYY-MM-DD` для запросов, `DD.MM.YYYY` для ответов. Несовместимы между собой, нужно конвертировать |
| ⏱️ | **Нет rate-limit, но реестр нагружать нельзя** — между запросами выдерживаем задержку `0.6 ± 0.1` секунды чтобы быть корректным клиентом |

## 🗄 Структура БД

Четыре таблицы PostgreSQL. Структура спроектирована так чтобы:
- каждый этап пайплайна писал в **свою** таблицу (не было пересечений ответственности),
- все операции были **идемпотентными** (повторный запуск не дублирует данные),
- ручные правки человека **сохранялись** между прогонами автоматики.

---

### `dim_brands` — справочник брендов

Список брендов которые мы решили отслеживать. Заполняется Шагом 0 на основе жёстко прописанного списка из `dimbrand.py`.

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | SERIAL | PK |
| `бренд` | TEXT UNIQUE | внутреннее название бренда (например, `Декаст`) |
| `поиск_mit` | TEXT | подстрока для запроса `GET /mit?search=...` — то что мы реально шлём в API |
| `ключевые_слова` | TEXT | варианты написания через запятую, по которым отсеиваем шум в результатах API |
| `страна` | TEXT | страна производителя из MIT detail |
| `город` | TEXT | город производителя из MIT detail |

---

### `manufacturers_verifiers` — производители по годам

Кто реально производит каждую модель в каждом году. Заполняется Шагом 1 — для каждого бренда находим его модели через MIT, потом для каждой модели смотрим VRI и определяем кто из поверителей с долей `vriType=1 ≥ 70%` — производитель.

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | SERIAL | PK |
| `reg_number` | TEXT | рег.номер модели — из `MIT.number` |
| `verifier` | TEXT | поверитель — из `VRI.org_title` |
| `year` | INTEGER | год — из `VRI.year` |
| `primary_count` | INTEGER | сколько первичных поверок (`vriType=1`) в выборке |
| `periodic_count` | INTEGER | сколько периодических (`vriType=2`) в выборке |
| `share_pct` | REAL | процент первичных от всех в выборке |
| `is_producer` | BOOLEAN | `TRUE` если `share_pct ≥ THRESHOLD` (70% по умолчанию) |

**Составной UNIQUE:** `(reg_number, verifier, year)` — одна и та же тройка может быть только в одной записи. То есть у одной модели в одном году могут быть **несколько разных поверителей**, и для каждого своя оценка «производитель или нет».

---

### `dim_reference` — характеристики моделей

Характеристики каждой модели: какие у неё бывают диаметры, к какой категории относится, теплосчётчик это или нет, и другие признаки. Заполняется Шагом 2.

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | SERIAL | PK |
| `рег_номер` | TEXT | рег.номер модели |
| `бренд` | TEXT | бренд из `dim_brands` |
| `наименование_си` | TEXT | название модели из `MIT.title` |
| `обозначение` | TEXT | обозначение из `MIT.notation` |
| `модификация` | TEXT | список диаметров через запятую: `15, 20, 25, 40` |
| `категория` | TEXT | категория счётчика (см. ниже) |
| `тип_счётчика` | TEXT | конкретный тип (крыльчатый, турбинный, ультразвуковой и т.д.) |
| `умный` | BOOLEAN | есть ли электронный вывод (M-Bus, импульсный) |
| `мокроходный` | BOOLEAN | мокроход (счётный механизм в воде) — для счётчиков воды |
| `многоструйный` | BOOLEAN | многоструйный — для счётчиков воды |
| `теплосчётчик` | BOOLEAN | да/нет — определяется по словам в наименовании |
| `источник` | TEXT | откуда взяли модификацию: `PDF` / `VRI` / `ручное` |
| `ручное_исправление` | TEXT | значение прописанное человеком вручную — имеет приоритет над автоматикой |

**Составной UNIQUE:** `(рег_номер, категория)` — у одной модели может быть **несколько строк**, по одной на каждую категорию. Например модель с DN15, DN20, DN25, DN40 даст две строки: одну с категорией «бытовые» (диаметры 15, 20), другую с категорией «домовые» (диаметры 25, 40).

**Возможные значения `категория`:**
- `Счетчики воды бытовые` — DN15-DN20 (квартирные)
- `Счетчики воды домовые` — DN25-DN40 (общедомовые узлы учёта)
- `Счетчики воды промышленные` — DN50+ (промышленные объекты)
- `Теплосчетчики` — отдельная категория, не делится по диаметру

### `production_volumes_exact` — объёмы поверок

Главная таблица — объёмы поверок по каждой модели в каждом месяце, с разбивкой по категориям. Источник данных для дашборда. Заполняется Шагом 4 (точная версия из `вольюмс.py`).

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | SERIAL | PK |
| `рег_номер` | TEXT | рег.номер модели |
| `организация_поверитель` | TEXT | поверитель |
| `бренд` | TEXT | бренд из `dim_brands` (денормализация для DataLens) |
| `наименование_си` | TEXT | название модели (денормализация) |
| `год` | INTEGER | год поверок |
| `месяц` | INTEGER | месяц (1..12) |
| `месяц_название` | TEXT | название месяца текстом (Январь, Февраль…) — для дашборда |
| `категория` | TEXT | категория счётчика (см. `dim_reference.категория`) |
| `количество` | INTEGER | число поверок данной (модель + поверитель + год + месяц + категория) |
| `возможные_диаметры` | TEXT | все диаметры модели через запятую — справочно |
| `точность` | TEXT | как посчитано (см. ниже) |
| `updated_at` | TIMESTAMPTZ | момент последнего обновления записи |

**Составной UNIQUE:** `(рег_номер, год, месяц, категория, организация_поверитель)` — это самый детальный уровень. Одна и та же модель может встретиться в нескольких строках за один месяц если поверителей у неё несколько или категорий несколько.

## 🏗 Архитектура

Пайплайн из четырёх этапов. Каждый этап читает результат предыдущего из БД и пишет в свою таблицу.

```
┌────────────────┐    ┌──────────────────────────┐    ┌──────────────────┐    ┌──────────────────────────┐
│  ШАГ 0         │    │  ШАГ 1                   │    │  ШАГ 2           │    │  ШАГ 3                   │
│  dimbrand.py   │ ─▶ │  Обновлпервич.py         │ ─▶ │  енрич.py        │ ─▶ │  вольюмс.py              │
│                │    │                          │    │                  │    │                          │
│  dim_brands    │    │  manufacturers_verifiers │    │  dim_reference   │    │  production_volumes_exact│
└────────────────┘    └──────────────────────────┘    └──────────────────┘    └──────────────────────────┘
        │                        │                            │                          │
        │                        │                            │                          ▼
        └────────────────────────┴────────────────────────────┴──────────────▶  📊 Yandex DataLens
                                                                                   (дашборды)
```

Сверху — оркестратор `запук.py`. Управляет очерёдностью, параметрами и флагами `RUN_*` для отдельных этапов.

## 🚀 Запуск

### Полный пайплайн (раз в полгода)

```bash
python запук.py
```

Запустит все этапы по очереди — от `dimbrand.py` до `вольюмс.py` для всех годов из `YEARS`.

### Только обновление объёмов (ежемесячно)

В файле `запук.py` поставить:

```python
RUN_DIMBRAND = False
RUN_PERVICH  = False
RUN_ENRICH   = False
RUN_VOLUMES  = True   # ← только этот шаг

YEARS = [2025]        # ← только текущий год
```

Дальше — `python запук.py`. Прогон будет в разы быстрее.

### Запуск отдельного шага

Любой скрипт можно запустить независимо:

```bash
python dimbrand.py
python Обновлпервич.py 2025 --pings 90 --positions 6 --threshold 70
python енрич.py
```
